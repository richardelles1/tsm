1. SYSTEM_OVERVIEW.md

(Purpose and product intent)

Explain what The Shared Mile actually is.

Example structure:

# System Overview

The Shared Mile (TSM) is a movement-triggered philanthropic marketplace.

Donors commit funds first.
Athletes unlock those funds through verified activity.

Core lifecycle:
Movement → Claim → Verification → Release → Payable

System goals:
- Trust and transparency
- Accurate financial ledger
- Clear athlete experience
- Verifiable philanthropic impact

This gives the AI the business context.

3. INVARIANTS.md

(Rules that must never break)

Example:

# System Invariants

1. Only one active claim per athlete.
2. Reserved claims expire after 90 seconds.
3. Funding pools cannot go negative.
4. Release amounts must reconcile with funding pools.
5. Payables must reference a release.
6. Financial integrity always overrides UI convenience.

These are the hard guardrails.

4. CLAIM_LIFECYCLE.md

(State machine for the core system)

Example:

# Claim Lifecycle

available → reserved → claimed → verified → released

available
Athlete can claim.

reserved
Temporary hold while athlete confirms.

claimed
Athlete commits to challenge.

verified
Activity is confirmed.

released
Funds are unlocked from funding pools.

This prevents AI from breaking the state model.

9. KNOWN_EDGE_CASES.md

(Things that look like bugs but aren’t)

Example:

# Known Edge Cases

Duplicate approved claims may appear during development testing.

Release engine is idempotent:
If a release already exists for a claim, no new release is created.

Funding pools must never go negative.

This prevents AI from breaking safety systems while “fixing” things.
